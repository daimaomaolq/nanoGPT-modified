# ModernKVCacheGRPO0.3 Plan

This document defines the v3 iteration for `nanoGPT-modified`: a small,
reproducible mini-GRPO alignment experiment built on top of the existing
modern nanoGPT codebase.

The goal is not to reproduce DeepSeek-R1. The goal is to implement and verify
the core mechanics of GRPO-style reinforcement learning alignment in a compact
nanoGPT setting:

- SFT warmup on automatically generated arithmetic prompts.
- Group rollout with multiple sampled responses per prompt.
- Rule-based rewards for answer correctness and output format.
- Group-relative advantage normalization without a critic.
- KL penalty against a frozen SFT reference policy.
- Evaluation with accuracy, reward, format pass rate, KL, and response length.

The v3 implementation should keep `model.py` stable unless a small shared helper
becomes clearly necessary. GRPO belongs in separate training scripts because it
has a different objective, batching shape, and logging contract than standard
next-token language modeling.

## Why Math Tasks

Mini-GRPO needs a task that is cheap to sample and easy to score automatically.
Arithmetic is a good first target because the final answer can be verified
without a learned reward model.

This is intentionally narrower than general reasoning alignment. To avoid a
thin evaluation story, v3 uses three evaluation layers:

1. Synthetic arithmetic holdout generated by this repo.
2. Public arithmetic word-problem benchmarks, such as GSM8K, MAWPS, or SVAMP.
3. Behavior metrics that catch reward hacking: format pass rate, invalid answer
   rate, response length, and KL drift.

## Baselines

The correct v3 baseline is not the v2 language-modeling validation loss. The
task changed from next-token pretraining to rule-reward alignment.

Required baselines:

| Baseline | Purpose |
| --- | --- |
| SFT-only checkpoint | Main baseline for GRPO improvement. |
| SFT-continued checkpoint | Controls for extra optimization steps and compute. |
| SFT + GRPO | Candidate v3 method. |

Recommended ablation:

| Ablation | Purpose |
| --- | --- |
| SFT + GRPO without KL | Shows whether KL stabilizes response behavior. |

Optional sanity checks:

| Check | Purpose |
| --- | --- |
| Random scratch model | Confirms the task is not solved by chance. |
| Shuffled reward | Confirms improvement depends on real reward signal. |

## Training Design

### Prompt and Response Format

All training and evaluation examples use a strict answer tag:

```text
Question: What is 17 + 24?
Answer: <answer>41</answer>
```

During SFT, the model sees the full prompt plus target response.

During GRPO, the model sees only the prompt and samples a response. The reward
parser extracts the first strict answer tag.

### Synthetic Data Curriculum

The initial implementation uses deterministic generated datasets:

| Stage | Operations | Digits |
| --- | --- | --- |
| easy | addition/subtraction | 0-9 |
| medium | addition/subtraction | 0-99 |
| hard | addition/subtraction/multiplication | 0-99 |

The default v3 experiment should start with `medium`, then optionally add
`hard` once the pipeline is stable.

### Tokenizer

The first implementation uses a repo-local character vocabulary for arithmetic
tasks. This avoids dependencies on GPT-2 BPE and keeps generated datasets tiny.

The vocabulary includes:

- Digits.
- Basic operators.
- English prompt characters used by templates.
- XML-like answer tags.
- Newline, padding, BOS, EOS.

This is not a general tokenizer. It is a controlled experimental vocabulary for
the mini-GRPO task.

## SFT Objective

SFT uses ordinary next-token cross entropy, but loss is applied only to response
tokens. Prompt tokens are masked with `-1`, matching the existing `GPT.forward`
loss convention.

Expected SFT outcome:

- High format pass rate.
- Reasonable accuracy on synthetic arithmetic.
- Stable checkpoint that can be copied as the GRPO reference policy.

## GRPO Objective

For each batch of prompts:

1. Sample `G` responses from the current policy.
2. Score every response with a rule-based reward.
3. Normalize rewards within each prompt group:

```text
advantage = (reward - group_mean) / (group_std + eps)
```

4. Compute response-token log probabilities under the current policy.
5. Compute reference log probabilities under the frozen SFT reference.
6. Optimize:

```text
loss = -mean(advantage * current_logprob_sum) + kl_coef * mean(token_kl)
```

The implementation is deliberately a small educational GRPO approximation. It
does not use a critic, generalized advantage estimation, or a learned reward
model.

## Metrics

Evaluation should report:

| Metric | Meaning |
| --- | --- |
| accuracy | Extracted answer exactly matches the gold numeric answer. |
| format_pass_rate | Response contains one strict `<answer>...</answer>` answer. |
| average_reward | Mean rule reward. |
| invalid_answer_rate | No parseable answer or malformed answer. |
| average_kl | Mean token-level KL proxy against reference policy. |
| average_response_length | Mean generated response length in tokens/chars. |

For training logs, also report:

- Samples per second.
- Mean group reward.
- Mean group reward standard deviation.
- Policy loss.
- KL loss.
- Total loss.

## Public Benchmarks

Recommended public benchmark order:

1. `MU-NLPC/Calc-mawps` for quick arithmetic word-problem sanity.
2. `MU-NLPC/Calc-svamp` for public test-only arithmetic.
3. `openai/gsm8k` for the main public benchmark.

The code should not require these datasets for local smoke tests. Public
benchmarks can be prepared on the server after dependencies and network access
are available.

## Server Experiment Flow

### 1. SFT smoke run

```bash
python grpo/train_sft.py \
  --out_dir out-grpo-sft-smoke \
  --device cuda \
  --dtype bfloat16 \
  --max_iters 200 \
  --eval_interval 50 \
  --batch_size 32 \
  --block_size 256 \
  --n_layer 4 \
  --n_head 4 \
  --n_embd 256 \
  --compile False
```

### 2. SFT full run

```bash
python grpo/train_sft.py \
  --out_dir out-grpo-sft \
  --device cuda \
  --dtype bfloat16 \
  --max_iters 3000 \
  --eval_interval 250 \
  --batch_size 64 \
  --block_size 256 \
  --n_layer 4 \
  --n_head 4 \
  --n_embd 256 \
  --norm_type rmsnorm \
  --mlp_type swiglu \
  --position_embedding_type rope \
  --compile False
```

### 3. SFT-only evaluation

```bash
python grpo/evaluate.py \
  --checkpoint out-grpo-sft/ckpt.pt \
  --split test \
  --device cuda \
  --dtype bfloat16 \
  --num_examples 1000 \
  --out_file out-grpo-sft/eval_test.json
```

### 4. SFT-continued baseline

```bash
python grpo/train_sft.py \
  --init_from out-grpo-sft/ckpt.pt \
  --out_dir out-grpo-sft-continued \
  --device cuda \
  --dtype bfloat16 \
  --max_iters 1000 \
  --eval_interval 250 \
  --batch_size 64 \
  --compile False
```

### 5. GRPO run

```bash
python grpo/train_grpo.py \
  --init_from out-grpo-sft/ckpt.pt \
  --reference_from out-grpo-sft/ckpt.pt \
  --out_dir out-grpo-v3 \
  --device cuda \
  --dtype bfloat16 \
  --max_iters 1000 \
  --eval_interval 100 \
  --batch_size 16 \
  --group_size 4 \
  --max_new_tokens 32 \
  --kl_coef 0.02 \
  --learning_rate 1e-5 \
  --compile False
```

### 6. GRPO evaluation

```bash
python grpo/evaluate.py \
  --checkpoint out-grpo-v3/ckpt.pt \
  --reference_checkpoint out-grpo-sft/ckpt.pt \
  --split test \
  --device cuda \
  --dtype bfloat16 \
  --num_examples 1000 \
  --out_file out-grpo-v3/eval_test.json
```

## Success Criteria

Minimum v3 success:

- SFT checkpoint reaches stable answer format.
- GRPO training runs without NaNs or uncontrolled KL growth.
- GRPO improves average reward or accuracy over SFT-only on held-out synthetic
  arithmetic.
- Format pass rate remains high after GRPO.

Stronger result:

- GRPO beats SFT-continued under comparable extra compute.
- GRPO improves or preserves public benchmark scores on MAWPS/SVAMP/GSM8K.
- No-KL ablation has worse KL drift, lower format pass rate, or longer malformed
  generations.

## Risks

| Risk | Mitigation |
| --- | --- |
| Reward hacking | Strict answer parser, length penalty, invalid-answer penalty. |
| KL explosion | Frozen reference, KL logging, `kl_coef`, gradient clipping. |
| Sampling cost | Small model, small group size, short max response length. |
| Public benchmark mismatch | Keep synthetic holdout as primary controlled eval. |
| Overclaiming | Report as mini-GRPO alignment, not R1 reproduction. |
